1. Mental model — two VMs, three lifecycles

Tonka manages exactly two named VMs:

NameRoleBuilt fromBuilt how often
tonka-base Frozen template. Holds dotfiles, brew packages, toolchains, Claude CLI. TONKA_BASE_IMAGE (Tart registry image) Once. Re-built only via tonka rebuild-base.
tonka-vm Your live workspace. Holds ~/repos/ and ~/projects/. tart clone tonka-base Once (on first command needing the VM). Re-cloned if deleted.

There are three creation flows that the script orchestrates, each living in a different function:

  • A create_base_vm — builds tonka-base from a clean Tart image.
  • B ensure_vm — clones & boots tonka-vm from tonka-base.
  • C sync_claude_settings — keeps the running VM in sync with the host on every command.
Most "tonka commands" implicitly run all three: ensure_vm → sync_claude_settings → do the thing. Knowing which phase a given line belongs to is the key to debugging.

3. Entrypoints — what triggers VM creation

Every command flows through main() at tonka:954. The relevant code paths for creation:

CommandPathEffect on VMs
tonka rebuild-basecmd_rebuild_base → create_base_vmForce-recreates tonka-base. Prompts to also delete tonka-vm.
tonka new-repo …cmd_new_repo → ensure_vm → ensure_base_vmCold path: creates both VMs the first time. Warm path: just boots tonka-vm.
tonka new … / tonka cl … / tonka pr …same cascadeSame: triggers full creation if either VM is missing.
tonka startcmd_start → ensure_vmIdentical cascade — useful for "warm up the VM".
tonka sh, tonka cp, tonka cursorall call ensure_vmSame cascade.
The cascade is: ensure_vm calls ensure_base_vm, which calls create_base_vm only when vm_exists tonka-base is false. So you never call create_base_vm directly except via rebuild-base.

4. Phase A — Base VM creation tonka:363–457 · create_base_vm()

Runs once. Builds the immutable template. Takes ~10–20 minutes (depends on brew package count).

  1. Pre-flight check — verify sshpass is installed; die if not. tonka:367–369
  2. Ensure SSH key — ensure_ssh_key() creates ~/.ssh/id_ed25519_tonka if missing (passphraseless, comment "tonka"). tonka:84–89, called from line 371
  3. Clone the macOS image — tart clone $TONKA_BASE_IMAGE tonka-base. First time this downloads ~10–20 GB from the Tart registry. tonka:375
  4. Configure resources — tart set applies TONKA_CPU, TONKA_MEMORY, and optionally TONKA_DISK_SIZE. Defaults: all host CPUs, ⅝ of host RAM, no disk override. tonka:376–381
  5. Boot the VM (headless) — tart run --no-graphics tonka-base & as a backgrounded child, PID saved in $vm_pid. Not disowned here (different from ensure_vm), so the script can wait $vm_pid at the end. tonka:384–385
  6. Initial boot wait — hard-coded sleep 30, then poll tart ip tonka-base via wait_for_vm_ip (60-attempt loop, 1s each). tonka:389–392, wait_for_vm_ip at 142–160
  7. SSH bootstrap with default creds — loop trying sshpass -p admin ssh admin@$ip true until success or 60s timeout. The macOS-Tahoe-Xcode image ships with admin/admin. tonka:395–405
  8. Collect host brew packages — brew list --formula and brew list --cask joined as space-separated strings. These are forwarded into the VM as env vars. tonka:409–414
  9. Copy payload to /tmp on guest — three sshpass scp calls push:
    • guest/install.sh → /tmp/install.sh
    • tonka public key → /tmp/tonka.pub
    • tonka private key → /tmp/tonka_key (yes — both keys, see warning)
    tonka:417–420
    The private key is copied into the VM so the guest user can act as you against GitHub via SSH if the dotfiles repo or some workflow needs it. This is fine for a single-tenant ephemeral sandbox, but worth knowing if you ever share the VM image.
  10. (Optional) copy Claude settings — if present locally, push ~/.claude/settings.json and the ~/.claude/skills tree to /tmp/. tonka:423–430
  11. Build install env — assemble single shell string with TONKA_USER, TONKA_DOTFILES_REPO, GITHUB_TOKEN, TONKA_TOOLS, BREW_FORMULAE, BREW_CASKS. Each only added if non-empty. tonka:433–449
  12. Run guest installer — sshpass ssh admin@$ip "chmod +x /tmp/install.sh && $install_env /tmp/install.sh". The bulk of the work happens inside this single SSH call. See Phase B. tonka:450
  13. Stop and freeze — tart stop tonka-base; wait $vm_pid to reap the backgrounded tart run. The template now exists on disk and will never be booted again until you rebuild-base. tonka:452–454

5. Phase B — Guest-side install.sh guest/install.sh

Runs as admin inside the base VM, drives all the heavy lifting that bakes into the template. Every sudo -u "$TUSER" call switches to the new tonka user.

  1. Create the work user — sysadminctl -addUser "$TONKA_USER" -fullName "Tonka User" -password tonka -admin. The username matches your host login so paths are familiar. install.sh:24–30
  2. Wire SSH keys — public key → authorized_keys; private key copy → ~/.ssh/id_ed25519; chown -R to the user; ssh-keyscan github.com into known_hosts. install.sh:33–50
  3. Passwordless sudo — drop $TUSER ALL=(ALL) NOPASSWD: ALL into /etc/sudoers.d/tonka. install.sh:54–55
  4. Install Homebrew — official installer with NONINTERACTIVE=1. Adds /opt/homebrew/bin to the script's PATH. install.sh:58–62
  5. Install essentials — brew install git gh jq. Hard requirement; failure aborts. install.sh:65
  6. Mirror host packages — brew install $BREW_FORMULAE || true, then brew install --cask $BREW_CASKS || true. The || true tolerates packages that don't apply (arch mismatches, etc.). install.sh:67–81
  7. Toolchains loop — for each token in TONKA_TOOLS: rust uses rustup; go/nodejs/python use brew. install.sh:83–105
  8. Install Claude CLI — curl -fsSL https://claude.ai/install.sh | bash. Lands at ~/.local/bin/claude. install.sh:108–109
  9. Seed Claude config — if the host pushed /tmp/claude_settings.json and/or /tmp/claude_skills, install them under ~/.claude/ with correct ownership. install.sh:112–123
  10. Enable Remote Login (SSH daemon) — systemsetup -setremotelogin on. This is what makes subsequent ed25519-key SSH from the host work. install.sh:126–127
  11. Git credential helper + (optional) gh auth — credential helper is always pinned to gh auth git-credential (so a token delivered later via Phase D's hosts.yml sync will work). If GITHUB_TOKEN was forwarded, also seed the guest's gh state via stdin-piped gh auth login --with-token so the very next step (dotfiles clone) can authenticate. install.sh:130–147
  12. Clone & run dotfiles — if TONKA_DOTFILES_REPO is set: convert SSH URLs to HTTPS (so the gh helper is used), git clone to ~/.dotfiles, then cd ~/.dotfiles && ./setup.sh. Errors here propagate. install.sh:151–191
  13. Cleanup — delete /tmp/tonka.pub, /tmp/tonka_key, /tmp/install.sh. install.sh:201–202

6. Phase C — Tonka VM creation tonka:197–225 · ensure_vm()

This is what runs every time you invoke a command that needs the VM. Most of it is the no-op fast path; the creation branch only fires the very first time.

  1. Cascade to base — ensure_base_vm calls create_base_vm iff tonka-base doesn't exist. This is the only place where Phase A normally fires from a non-rebuild command. tonka:198 → 190–195
  2. Clone if missing — when tonka-vm doesn't exist: tart clone tonka-base tonka-vm, then tart set with the host CPU/RAM (re-computed per host) and an optional --disk-size. tonka:199–208
    The clone copies the disk image — fast (CoW on APFS) but the new VM gets a different MAC, so it'll get its own IP on boot.
  3. Boot if stopped — when tonka-vm isn't running: tart run --no-graphics tonka-vm &, then disown so the script can return. sleep 15 as initial wait. tonka:209–212
  4. Discover IP — wait_for_vm_ip polls tart ip tonka-vm up to 60 times (1s each). tonka:213–215
  5. Wait for SSH — wait_for_ssh tries ssh -i id_ed25519_tonka $TONKA_USER@$ip true in a BatchMode loop with StrictHostKeyChecking=no; 60 attempts max. tonka:216, 162–174
  6. Update ~/.ssh/config — update_ssh_config: if the file doesn't exist, prompts to create it; if no Host tonka block, prompts to add it; if the block exists with a different IP, rewrites the HostName in place with sed. tonka:217, 91–135
  7. Fast path on warm runs — if the VM was already running, skip clone/boot/IP-wait but still call update_ssh_config with the current IP to fix stale entries. tonka:219–224
Edge cases worth knowing
  • The sleep 15 at line 212 is shorter than the sleep 30 in create_base_vm — because Phase C uses ed25519 key auth (faster to retry on failure) while Phase A still uses sshpass.
  • If wait_for_vm_ip times out the script dies with no rollback — the half-created tonka-vm remains and a re-run will hit the "exists" branch and try to start it again.
  • The tart run … & child is disowned, not waited-on. The VM keeps running after the script exits. That's the design — long-lived background VM.

7. Phase D — sync_claude_settings tonka:227–361

Runs on most commands (new-repo, new, pr, cl) right after ensure_vm. Idempotent: md5-compares each artifact before re-uploading.

  1. Ensure ~/.claude exists on guest.tonka:252
  2. settings.json — md5 host vs guest; scp if changed.tonka:255–263
  3. skills/ — recursive md5 (sort-then-hash) host vs guest; scp -r if changed.tonka:266–275
  4. gh hosts.yml — md5 compare; scp if changed.tonka:277–287
  5. git credential helper — assert helper is gh auth git-credential, rewriting if not. Clears any osxkeychain default that Xcode might've installed.tonka:290–293
  6. Plugins — read ~/.claude/plugins/installed_plugins.json on host, add the official marketplace, install each plugin not already present.tonka:296–316
  7. Onboarding flag — ensure ~/.claude.json contains hasCompletedOnboarding: true so Claude doesn't open the first-run wizard.tonka:319–328
  8. CLAUDE.md — md5 compare; scp if changed.tonka:331–339
  9. Keychain credentials — security find-generic-password -s "Claude Code-credentials" on host; md5 against ~/.claude/.credentials.json on guest; piped via stdin to ssh "cat > … && chmod 600 …" if changed (quote-safe; file is mode 600 on the guest).tonka:342–355
The credential write now uses a stdin pipe (printf '%s' "$creds" | ssh ... "cat > … && chmod 600 …") so the blob is never interpolated into a shell string — quote-safe regardless of content, and the destination file is mode 600.
The skills hash uses find -L so a symlinked ~/.claude/skills (e.g. into a dotclaude-staging repo) is traversed correctly. Without -L, BSD find would skip the top-level symlink and the hash would collapse to md5-of-empty.
On entry, sync_claude_settings warns once if neither $GITHUB_TOKEN nor gh auth status succeeds on the host — surfacing the most common silent-failure mode for VM git operations. It also warns about broken symlinks under ~/.claude before the sync runs.

7b. gh authentication lifecycle

Git auth inside the VM follows two paths depending on remote shape:

Remote shapePathSource of truth
HTTPSguest's gh auth git-credential helper reads ~/.config/gh/hosts.ymlHost's hosts.yml, resynced via Phase D every command
SSH (git@…, ssh://…)SSH agent forwarding (ssh -A) from host into VM during the git operationHost's running ssh-agent
Dotfiles clone in Phase BSSH URLs are rewritten to HTTPS by ssh_to_https(); uses the gh credential helperThe GITHUB_TOKEN forwarded into install.sh, or — if absent — the helper waits for Phase D to deliver a hosts.yml

The credential helper is now always configured on the guest (install.sh:130–138), regardless of whether the host had a token at base-build time. This means a fresh base VM built without GITHUB_TOKEN still gets a working helper, and Phase D delivers the actual token on the first command after the host runs gh auth login.

Use tonka auth-status to print the current host + guest auth state side by side.

8. Phase E — Creating a project tonka:492–520 · cmd_new

The user-facing "make me a Claude session" command. Everything above already ran by this point.

  1. ensure_vm — Phase C cascade.tonka:499
  2. sync_claude_settings — Phase D.tonka:500
  3. Default repo pick — if no repo arg, ls ~/repos | head -1 on the VM; die if empty.tonka:505–511
  4. Worktree creation — single SSH with -A (agent forwarding): 
    mkdir -p ~/projects
cd ~/repos/$repo && git fetch origin main
git worktree add ~/projects/$project -b $project origin/main
cd ~/projects/$project && git push -u origin $project
    tonka:513–515
  5. Launch Claude — interactive SSH (-t) that cds into the worktree, starts an interactive login shell, runs ~/.local/bin/claude --dangerously-skip-permissions, and on exit drops you into a fresh shell with a hint: 
    Claude exited. You are still in the tonka VM.
Press Ctrl-D to leave.
    tonka:517–519
cmd_pr (line 498) follows the same shape, but resolves a PR's branch via gh pr view, then either re-enters the existing worktree or creates a new one named after the branch.
